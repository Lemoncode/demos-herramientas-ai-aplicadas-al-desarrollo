# Four-phase orchestration with measurable stop and dedicated Review phase

The `/goal` command needs an explicit, audience-visible stop condition — "all 6 Agent calls returned" is not enough because a Worker can return "success" with broken tests, missing a11y, or sloppy code. We considered (i) per-Worker self-review (each Worker invokes accessibility + 4R checks internally before returning) and (ii) running Reviewers on a merged main branch after the Fleet; we rejected both because the former hides the Reviewers from the demo's parallel-agents narrative and the latter requires a merge step that doesn't exist in the demo flow.

We adopted a four-phase pipeline: **Foundation** (Orchestrator alone, sequential, ends in a `foundation` commit), **Fleet** (6 Worker subagents in parallel, each in an `Agent(isolation: "worktree")` worktree), **Review** (2 read-only Reviewer subagents — `accessibility-reviewer` and `4r-reviewer` — each dispatched once with the 6 worktree paths), and **Ship & Report** (Orchestrator opens one PR per Section whose `build_pass && tests_pass`, then prints a 6×3 pass/fail matrix as the Final Report). Printing the Final Report is the explicit stop condition; the Orchestrator does not exit until that matrix is on screen.

Consequence: `/goal` is no longer side-effect-free — it pushes branches and opens PRs against `origin`. The harness now depends on a configured GitHub remote and authenticated `gh` CLI before `/goal` can run. Re-running `/goal` on an unreset main will produce duplicate or conflicting branches; the operator is expected to reset main between live demos. Blocked Sections (build or test failure) get **no PR** — broken code never reaches GitHub — and appear as `—` in the report's PR column.
