Basic
Concepts

01 — Tokens

What is a token

  • "Hello" → can be 1 or several tokens depending on the LLM
  • "🚀" → emojis can be one or several several tokens
  • Spaces, accents, and symbols also count
  • Most LLM's take spaces as part of the next / previous word
  • Each model can tokenize differently

Key idea: the model processes statistical fragments, not words.

openai/tiktoken — Byte-pair encoding tokenizer
Wikipedia — Byte-pair encoding

02 — Token Estimation

A token is not
always a word

  • 1 short sentence ≈ 10–20 tokens
  • 1 book page ≈ 500–800 tokens
  • 1 blog post ≈ 1,500–3,000 tokens
  • 1 full documentation page with code ≈ 3,000–8,000 tokens
  • 1 small source file ≈ 1,000–3,000 tokens
  • 1 large source file ≈ 8,000–20,000+ tokens
  • 100,000 tokens ≈ a small book, several long docs...
03 — Temperature

How random
is the output

Temperature controls how the model samples the next token. Low values make output deterministic; high values make it creative — and unpredictable.

Claude and Gemini models have a default temperature of 1

T = 0 · Deterministic

Always picks the most probable token. Reproducible across runs.

Code, math, factual Q&A, structured extraction

T ≈ 1 · Balanced

Samples from the natural probability distribution. Default for most APIs.

Conversation, summaries, general writing

T > 1 · Creative

Boosts low-probability tokens. Output becomes diverse — risks incoherence.

Brainstorming, creative writing, ideation

04 — Vector databases

Words as
vectors

Each word or concept is represented as a point in high-dimensional space. Semantically similar concepts end up geometrically close — enabling arithmetic on meaning.

  • king − man + woman ≈ queen
  • Distance in vector space = semantic similarity
  • A vector DB stores and queries these points at scale
05 — Semantic search

Semantic
search

Text is converted into numerical vectors (embeddings) that capture meaning. Similar concepts cluster nearby in high-dimensional space.

  • An embedding model maps text → a vector of floats
  • Similarity is measured with cosine distance
  • Vector DBs index millions of vectors for fast nearest-neighbour search
  • Retrieval by meaning, not exact keywords

Mikolov et al. — Efficient Estimation of Word Representations (2013)
Illustrated Word2Vec — Towards Data Science

Word2Vec analogy
king − man + woman ≈ queen

vectors encode semantic relationships

Similarity search
query → [0.21, −0.83, 0.54, ...]
cosine_similarity(query, docs)
→ top-k nearest neighbours
<Module 02/>

Context
Window

01 — What fills it

Context
window

The context window is the maximum number of tokens the model processes in a single call — everything it "sees" before generating the response.

  • Includes prompt, history, documents, and tool results
  • It is not persistent memory — it resets every conversation
  • When it fills up: the model has to decide what to keep and what to discard

Note: active plugins and MCP tools can add extra context.

Key idea: the model does not remember — it only processes what fits in the current window.

02 — Position effects

Lost
in the middle

LLMs process context unevenly. Information at the start and end gets preferential attention — the middle is systematically underweighted.

Liu et al. — Lost in the Middle (Stanford, 2023)
Zhao et al. — Serial Position Effects in LLMs (2024)

▲ Primacy bias

Strong recall of content at the start of context

◆ Middle

Critical details here are frequently missed or ignored

▼ Recency bias

Strong recall of content at the end of context

03 — Chain of thought

Engineer
position

  • Put the task objective and key constraints at the start of your prompt
  • Place format instructions and final reminders at the end
  • Never bury critical details in the middle of large context dumps
  • Summarize long documents — avoid injecting full files inline
  • Prompt-based mitigation helps but is inconsistent — always verify empirically

Key idea: context position is a design decision — treat it as intentionally as word choice.

Liu et al. — Lost in the Middle (Stanford, 2023)
Zhao et al. — Serial Position Effects in LLMs (2024)

<Module 03/>

Compaction
Techniques

01 — The problem

When context
fills up

  • Context window has limits, overflow causes truncation or errors
  • Every accumulated token adds cost and latency
  • Irrelevant old content (And memory!) dilutes the model's attention on what matters
  • Agents fill the context window fast: tool calls, responses, history all stack up

Compaction is not optional in long-running agents — it is a core part of the architecture.

02 — Strategies

Four ways
to compact

  • Summarization — replace full history with a dense summary
  • Rolling window — keep only the last N messages
  • Selective pruning — drop low-value or irrelevant turns
  • RAG on history — retrieve only what this turn needs

Pick the strategy to match the task — conversations, long agents, and document tasks each need a different approach.

03 — Best practices

Compact
with intent

  • Compact before a new subtask, never mid-task
  • Include explicit summaries of decisions already made
  • Keep critical constraints and architecture choices in the summary
  • Use persistent memory for facts that must survive compaction
  • Test compaction regularly, do not wait for overflow to find out what breaks

    • Use /handoff to compact a conversation and delegate it to a sub-agent

01 — AGENTS.md / CLAUDE.md / GEMINI.md ...

A README
for agents

A file at the repo root that tells the agent how to work on your project. It complements README.md it does not replace it.

  • Compatible with Copilot, Cursor, Devin and more (Not with Claude / Gemini ATM)
  • Supported in more than 60,000 open source projects
  • Open standard by Linux Foundation / Agentic AI Foundation

BE AWARE: it is the context the agent loads on each request remember thatevery line you add has a token cost.

02 — AGENTS.md (Proxy style / Auto-invoke)

Proxy
style

The file does not instruct — it routes. Every rule, skill, and prompt is declared here but lives and executes elsewhere.

Use when: conventions already live in other files and tools.

03 — AGENTS.md (Instruction style)

Instruction
style

The file is self-contained. All rules, constraints, and workflow steps live here. The agent has everything it needs without following any references.

  • ✅ Explicit: no ambiguity about what the agent must do
  • ⚠️ Higher token cost: loaded in full on every request
  • ❗️ Risk of bloat: keep it focused, prune what is not agent-relevant

Use when: the agent needs specific domain knowledge or constraints that are not captured anywhere else. 

# AGENTS.md

## Code style
TypeScript strict mode. No `any` types.
Functions max 40 lines. No default exports.

## Workflow

2. Never push directly to main
3. Open a PR — always request review

## File structure

- Validate all input with zod at boundaries

05 — Anti-patterns

Most of this
doesn't belong

Hook

Lint, format and test commands should run automatically via a pre-commit hook — not as text the agent has to read every request.

Skill

Step-by-step workflows like "adding a component" are better as a reusable /skill the agent can invoke on demand.

Instructions

Test conventions belong in a dedicated instructions file (e.g. @docs/testing-conventions.md) — referenced from AGENTS.md, not embedded inline.

06 — Instructions structure

Same concept,
different syntax

Path-specific rule files use a frontmatter block to narrow context to a file pattern. The keyword is the only thing that changes between tools.

paths: src/components/**        ← Claude Code

Skills

01 — What they are

Invocable
instructions

  • Step-by-step instructions for repeatable tasks
  • In some IDE's you can invoke them using /skill-name
  • Only consume tokens when active
  • They can be project-specific or reusable across projects
02 — How they work

Structure of
a skill

  • Folder with the name of the skill containing a SKILL.md file in {X}/skills/
  • Frontmatter: name and description
  • Body: instructions in natural language
  • Steps: numbered instructions
  • Constraints (optional): rules and limitations
  • Checklist (optional): task-specific checklists

TIPS

  • Have in mind the freedom level you want to give the agent to do a certain task.
  • Skills can be chained. Think if having atomic skills is beneficial for your use case.

03 — Token savings

context:
fork

Runs the skill in an isolated subagent — the main conversation context is not consumed. The subagent gets only what the skill provides.

Use it when: the task is self-contained and the result can be handed back as a single output.

The best rule of thumb

Ask this:
Could a fresh assistant complete this task using only the SKILL.md content, its tools, and maybe command output?

If yes → use context: fork.

Ask this:
Do I want these instructions to guide the main agent while it codes, reasons, edits, or talks with me across turns?

If yes → do not use context: fork.

<Module 07/>

Prompts
and Commands

01 — What's a prompt?

Trigger a workflow by demand

  • A prompt is a request to the agent to perform a task or answer a question
  • It can be a single instruction, a multi-step workflow, or a complex set of constraints
  • IMPORTANT: They don't take space in the agent's context they are only used when invoked

Always include

  • Relevant role or context
  • Concrete, scoped task
  • Include examples

Could be good to include

  • Mention to use existing tools if available: Skills, MCPs, CLIs, etc.
  • Similar good practices as skills: frontmatter, constraints, checklist...

Avoid

  • Ambiguous requests
  • Multiple goals in one
  • Irrelevant context
  • Expecting the model to guess

A clear prompt reduces iterations — and saves tokens.

02 — Slash commands

Default slash commands ( / )

  • /clear — clear current context
  • /plan — create an implementation plan
  • /compact — manually compact context
  • /model — switch the active model
  • ...

Commands are shortcuts to complex workflows a slash instead of a paragraph of instructions.
